## 1. Why Run LLMs on CPU?

The honest answer: sometimes you have no choice. But "no choice" has quietly become "surprisingly capable" in the past two years.

A **3B parameter model** quantized to Q4_K_M fits in **~2 GB of RAM** and runs at 15–30 tokens per second on a modern CPU. A 7B model in the same format uses ~4.5 GB and delivers 8–15 tok/s with AVX-512. That's enough for chatbots, document summarization, code assistance, and simple agent loops.

The key enabler is **GGUF** — a file format developed by the llama.cpp project that packs model weights in a compressed, memory-mapped format optimized for CPU SIMD instructions.

**Three things make CPU inference viable today:**

1. **Quantization** — reducing weight precision from FP16 to 4–8 bits cuts RAM requirements by 2–4× with minimal quality loss
2. **SIMD acceleration** — modern CPUs (AVX2, AVX-512, AMX) handle matrix multiplications efficiently enough for real-time inference
3. **KV cache quantization** — compressing the attention cache reduces memory bandwidth pressure per generated token

## 2. CPU vs GPU: Why Does It Matter for LLMs?

Before diving into the code, it's worth understanding *why* GPU inference is faster — because it also explains why CPU inference isn't as bad as you might think for small models.

### 2.1 The Core Difference: Many Simple Cores vs Few Powerful Cores

A modern CPU has **8–32 powerful cores**, each capable of handling complex, sequential tasks: branch prediction, out-of-order execution, floating-point operations with deep pipelines. These cores are designed for general-purpose work — running an operating system, executing a web server, managing memory.

A GPU has **thousands of small, simple cores** (CUDA cores on NVIDIA, shader units on AMD). Each core is far less capable than a CPU core individually — it can't handle complex control flow or branching. But they all run **in parallel**, making GPUs ideal for operations that can be split into thousands of identical, independent computations.

LLM inference — multiplying large weight matrices by input vectors — maps perfectly onto GPU parallelism.

### 2.2 The Real Bottleneck: Memory Bandwidth

The gap isn't actually about compute. Modern CPUs are powerful enough to do the math. The gap is about **how fast you can move data from memory to the processor**.

| Hardware | Memory type | Bandwidth |
|---|---|---|
| GPU (NVIDIA H100) | HBM3 VRAM | ~3,350 GB/s |
| GPU (RTX 4090) | GDDR6X VRAM | ~1,008 GB/s |
| CPU (Intel i9-13900K) | DDR5 RAM | ~89 GB/s |
| CPU (Intel i7-1265U, laptop) | DDR4 RAM | ~51 GB/s |

LLM inference is **memory-bandwidth bound**: for each generated token, the processor needs to read *all* the model weights from memory. A 7B Q4_K_M model = ~4.5 GB of weights read every token.

- On an RTX 4090: 4.5 GB / 1,008 GB/s ≈ **4.5 ms/token → ~220 tok/s**
- On a laptop CPU: 4.5 GB / 51 GB/s ≈ **88 ms/token → ~11 tok/s**

That's roughly a **20× gap** — not because the CPU can't compute, but because it can't feed data fast enough.

### 2.3 Why CPU Inference Is Still Useful

The bandwidth gap is real, but it doesn't make CPU inference irrelevant:

- **No GPU available** — corporate laptops, edge servers, Raspberry Pi
- **Small models (1.5B–3B)** — the bandwidth gap shrinks; 1.5B at Q4_K_M is ~0.8 GB → even laptop RAM delivers 30+ tok/s
- **Privacy** — data never leaves the machine; no API calls, no telemetry
- **Cost** — zero inference cost after the initial hardware

The sweet spot: **3B instruct models on AVX2 CPU** deliver interactive-speed responses (~15 tok/s) with no GPU required.

## 3. Choosing Your Model

### 3.1 Instruct vs. Base Models for Agents

**Always use instruct models for agents.**

Base models are trained to predict the next token in a document — they don't understand the concept of "following an instruction." Instruct models go through a second phase: **Supervised Fine-Tuning (SFT)** on instruction/response pairs, often followed by RLHF or DPO alignment. They understand:

- The difference between a system prompt and a user message
- How to format structured outputs (JSON, code, lists)
- When to stop generating
- How to use tool call schemas

For an agent that needs to reliably route queries, call tools, and maintain conversation state, an instruct model is not optional — it's the foundation.

### 3.2 A New Architecture: Mixture of Experts (MoE)

Most models in this list are **dense** — every parameter is used for every token. A newer architecture, **Mixture of Experts (MoE)**, works differently.

In a MoE model, the feed-forward layers are split into multiple independent "expert" sub-networks. For each token, a small **router** selects only 2–4 of those experts to activate. The rest stay idle.

```
Dense model:  token → ALL parameters → output
MoE model:    token → router → 2 of N experts activated → output
```

**The key benefit:** you get the quality of a large model (many total parameters = rich knowledge) at the inference cost of a small model (few active parameters = low compute per token).

Two examples in this list use MoE:

**Qwen3-4B-Thinking** — part of the Qwen3 family which includes MoE variants. The `-Thinking` models add an explicit reasoning chain (`<think>...</think>`) before the final answer, inspired by DeepSeek-R1. This is separate from MoE but often combined with it in the Qwen3 architecture.

**Gemma 4 E4B** (`google/gemma-4-E4B-it`) — Google's latest MoE model. The name means **Effective 4B**: ~4B parameters are *active* per token, but the total model has far more parameters across all experts. This gives it reasoning quality far above a typical 4B dense model, while keeping CPU inference fast since only 4B params are loaded per forward pass. It supports a 128K context window and Google's multimodal training.

> **MoE on CPU:** MoE models load all expert weights into RAM (so total RAM needed = full model size), but only compute on 2–4 experts per token. This means lower CPU time per token but higher RAM usage compared to a dense model of the same active size. For CPU deployment, make sure you have enough RAM for the full model.

### 3.3 Best Small Models (Under 8B active params) for CPU in 2025

| Model | Size (active) | Architecture | Strengths | HuggingFace ID |

## 5. Quantization with `cpu_optimizer`

### 5.1 Quantization Profiles

Instead of choosing raw quantization types (Q4_K_M, Q5_K_M, etc.), the `cpu_optimizer` class exposes three simple profiles that mirror how you'd call `main_optimizer.py` from the command line:

```python
QUANT_PROFILES = {
    "precision": "Q6_K",    # best quality, more RAM
    "balanced":  "Q4_K_M",  # good tradeoff — default
    "speed":     "Q3_K_M",  # fastest, less RAM, quality reduced
}
```

> **Note on bit counts:** K-quant formats (Q4_K_M, Q5_K_M, Q6_K) use hierarchical super-blocks where some critical layers (attention, FFW) are promoted to higher precision. Q4_K_M is actually **~4.89 bits/weight** on an 8B model, not exactly 4. The perplexity column below uses WikiText-2 benchmarks from [arXiv 2601.14277 (2025)](https://arxiv.org/abs/2601.14277).

| Profile | Type | Bits réels (8B) | RAM (3B) | RAM (7B) | Perplexité (WikiText-2) | Quality |
|---|---|---|---|---|---|---|
| precision | Q6_K | 6.56 | ~2.9 GB | ~6.1 GB | ~7.35 | Excellent |
| **balanced** | **Q4_K_M** | **4.89** | **~2.0 GB** | **~4.6 GB** | **7.56** | **Good — recommended** |
| speed | Q3_K_M | 3.74 | ~1.6 GB | ~3.7 GB | ~8.0 | Degraded |
| — | Q8_0 | 8.50 | ~3.3 GB | ~7.9 GB | 7.33 | Near-identical to FP16 |
| — | F16 (baseline) | 16.00 | ~6.5 GB | ~14.9 GB | 7.32 | Reference |

Source: [llama.cpp quantize README](https://github.com/ggml-org/llama.cpp/blob/master/tools/quantize/README.md) · [arXiv 2601.14277](https://arxiv.org/abs/2601.14277)

### 5.2 How Auto-Selection Works

When you pass `quant_type="auto"`, the class reads your system:

```
RAM available → estimate model size per quantization type
CPU flags     → score SIMD capabilities (AMX > AVX-512VNNI > AVX-512 > AVX2)
→ pick the highest quality that leaves enough RAM headroom
```

If the system can't fit Q4_K_M it falls back to Q3_K_M, always warning you when quality will be noticeably degraded.

### 5.3 Running the Optimizer

```python
from cpu_optimizer import cpu_optimizer

QUANT_PROFILE = "balanced"
QUANT_TYPE    = QUANT_PROFILES[QUANT_PROFILE]

optimizer = cpu_optimizer(
    model_name="mistralai/Ministral-3B-Instruct-2412",
    input_path="./models/full/",
    output_path="./models/gguf/",
    quant_type=QUANT_TYPE,
    quantization_verbose=False,
)
optimizer.llama_dir = Path("./models/llama.cpp")

success, quant_used = optimizer.main_optimize()
```

The optimizer handles everything: cloning llama.cpp, compiling with the right CMake flags for your CPU (e.g., `-DGGML_AVX512=ON`), converting HF → GGUF F16, quantizing, and cleaning up the intermediate file.

## 6. Loading and Running Inference

With the GGUF file ready, we load it directly with `llama-cpp-python`:

```python
cores_p   = psutil.cpu_count(logical=False)
ram_avail = psutil.virtual_memory().available / 1024**3
n_ctx     = 4096 if ram_avail > 20 else 2048

try:
    model = Llama(
        model_path=str(optimizer.gguf_quantized),
        n_threads=cores_p,
        n_threads_batch=cores_p,
        n_ctx=n_ctx,
        n_batch=512,
        use_mmap=True,
        use_mlock=False,
        verbose=False,
        flash_attn=True,   # reduces memory for long contexts (llama.cpp >= b3770)
        type_k=8,   # Q8_0 KV cache — ~50% RAM reduction, near-zero quality loss
        type_v=8,   # use type_k=2/type_v=2 (Q4_0) for even more savings if RAM is critical
    )
except (TypeError, ValueError):
    model = Llama(model_path=str(optimizer.gguf_quantized),
                  n_threads=cores_p, n_ctx=n_ctx, verbose=False)
```

**Key parameters explained:**

- `n_threads=cores_p` — use **physical cores only** (not logical/hyperthreaded). HT doesn't help with LLM matrix operations and can hurt. This is #1 cause of slow inference per official llama.cpp docs. Source: [llama.cpp Performance Tuning](https://www.mintlify.com/ggml-org/llama.cpp/advanced/performance-tuning)
- `flash_attn=True` — reduces attention memory for long contexts (requires llama.cpp >= b3770)
- `type_k=8, type_v=8` — Q8_0 KV cache quantization: ~50% RAM reduction, near-zero quality loss. Use `type_k=2, type_v=2` (Q4_0) for even more savings when RAM is critical
- `use_mmap=True` — memory-maps the file; the OS loads pages on demand, making startup faster

### 6.1 Generating Text

```python
def generate(model, prompt, max_tokens=150):
    cpu_samples = []
    stop_flag   = threading.Event()

    def _monitor():
        while not stop_flag.is_set():
            cpu_samples.append(psutil.cpu_percent(interval=0.2))
    threading.Thread(target=_monitor, daemon=True).start()

    t0   = time.time()
    resp = model.create_chat_completion(
        messages=[{"role": "user", "content": prompt}],
        max_tokens=max_tokens,
        temperature=0.3,
        top_p=0.9,
        repeat_penalty=1.1,
    )
    elapsed   = time.time() - t0
    stop_flag.set()

    usage = resp["usage"]
    tok_s = usage["completion_tokens"] / elapsed

    return {
        "text":    resp["choices"][0]["message"]["content"],
        "tok_s":   round(tok_s, 1),
        "tokens":  usage["completion_tokens"],
        "time_s":  round(elapsed, 2),
        "cpu_pct": round(sum(cpu_samples) / len(cpu_samples), 1) if cpu_samples else 0,
    }
```

### 6.2 Sample Output

```python
result = generate(model, "Explain what a Large Language Model is in 3 sentences.")

print(result["text"])
# → A Large Language Model is a neural network trained on...

print(f"{result['tok_s']} tok/s  |  {result['tokens']} tokens  |  CPU {result['cpu_pct']}%")
# → 18.4 tok/s  |  87 tokens  |  CPU 94.2%
```

## 7. Benchmarking

Running a few prompts and measuring tok/s gives you a realistic picture of your hardware:

```python
prompts = [
    "What is quantization in machine learning?",
    "Write a Python function to sort a list of dictionaries by a key.",
    "What are the main advantages of running LLMs locally?",
]

rows = [generate(model, p, max_tokens=100) for p in prompts]

print(f"{'Prompt':<55} {'tok/s':>6} {'tokens':>7}")
print("-" * 70)
for p, r in zip(prompts, rows):
    print(f"{p[:54]:<55} {r['tok_s']:>6} {r['tokens']:>7}")

avg = sum(r['tok_s'] for r in rows) / len(rows)
print(f"\nAverage: {avg:.1f} tok/s")
```

**Performance expectations (Q4_K_M, with correct CMAKE_ARGS + OpenBLAS build):**

> ⚠️ These numbers assume `llama-cpp-python` was compiled with the AVX2/OpenBLAS flags above. Without them, expect 10–50× slower.

| CPU | Ministral-3B | Qwen2.5-7B |
|---|---|---|
| AVX2 only (most Intel/AMD laptops) | 12–20 tok/s | 5–10 tok/s |
| AVX-512 (Xeon, some Core) | 20–35 tok/s | 10–18 tok/s |
| AMX (Xeon 4th gen+, SPR/EMR) | 35–55 tok/s | 20–35 tok/s |

Source: [llama.cpp quantize README](https://github.com/ggml-org/llama.cpp/blob/master/tools/quantize/README.md)

### Real-world benchmark: Intel Core Ultra 9 285K on Linux

To validate these numbers, I tested the full pipeline on a desktop workstation running a Docker container with llama.cpp compiled from source:

```
CPU    : Intel Core Ultra 9 285K (24 physical cores, AVX2)
RAM    : 62 GB DDR5
OS     : Linux (Docker container, Ubuntu 22.04)
Model  : Ministral-3B-Instruct-Q3_K_M
Config : n_threads=24, flash_attn=True, ctx=71168, type_k=2, type_v=2
Result : 44.4 tok/s (eval) — 1382.3 tok/s (prompt processing)
```

**44.4 tokens per second on CPU — fast enough for real-time streaming in any application.**

A few observations from this test:

**Thread count is the most impactful parameter after SIMD.** Using all 24 physical cores (minus 1 for the OS) produced the full 44.4 tok/s. The same machine with 2 threads would deliver ~4–5 tok/s. This is why `get_inference_threads()` in `utils.py` defaults to `physical_cores - 1`.

**CPU generation matters less than setup quality.** This result was achieved with **AVX2 only** — no AVX-512 — because the Core Ultra 9 285K (Arrow Lake) does not support AVX-512. The 44.4 tok/s comes from 24 well-utilized threads + a properly compiled binary, not from exotic SIMD. An older Xeon with AVX-512 but only 8 cores would be slower.

**Linux outperforms Windows for CPU inference.** On an equivalent Windows laptop (i7-1265U, 10 cores, AVX2), the same model with the pre-built `cpu` wheel (SSE2 only) and 2 threads produced **2.9 tok/s** — a 15× gap. The factors in order of impact:

| Factor | Linux i9 | Windows i7 | Ratio |
|---|---|---|---|
| Threads | 24 | 2 | 12× |
| SIMD build | AVX2 (compiled) | SSE2 (pre-built cpu wheel) | ~4× |
| Code path | llama-cli binary | llama-cpp-python FFI | ~1.3× |
| OS scheduler | Linux | Windows (hybrid CPU) | ~1.3× |

Switching from the `cpu` wheel to the `avx2` pre-built wheel and using all threads would close most of that gap, pushing the i7 to an estimated **15–25 tok/s** — entirely usable for interactive applications.

**Q3_K_M on a fast CPU can beat Q4_K_M on a slow one.** If your CPU is well-configured (AVX2+, 8+ cores), the bandwidth savings from Q3_K_M outweigh the quality cost. A 44 tok/s Q3_K_M response *feels* better than an 8 tok/s Q4_K_M response, even if the latter is technically more accurate.

The key takeaway: **CPU inference is not a compromise — it is a legitimate deployment target.** With the right setup (Linux, AVX2 wheel, all physical cores), a modern desktop CPU delivers speeds that are perfectly usable for interactive chatbots, document summarization, and agent loops.
